# NucleoOS Internationalization (i18n)

NucleoOS ships bilingual — **Italian (primary) + English** — through one centralized runtime. This
document is the contract: how the system is wired, how to translate a screen, and how to add a
language. It is designed to scale (more languages, more apps) without touching the engine.

## Why centralized

The OS is a desktop **shell** plus ~40 **apps**, each loaded in an iframe from `/apps/<id>/`. Before
this system there were three incompatible patterns (a per-app string table in Settings, inline
`en ? '…' : '…'` ternaries in onboarding/copilot, and hardcoded Italian in the shell). One engine
replaces all of that: a single language signal, one persistence path, one way to translate a string.

Everything runs **in the browser**. The Cardputer only serves small static JSON catalogs, so the
device cost is ~0 — consistent with the no-PSRAM, web-native design.

## The pieces

| File | Role |
|------|------|
| `web/shell/nucleo-i18n.js` | The engine. Served at the OS root → importable as `/nucleo-i18n.js` from the shell **and every app iframe** (same origin). |
| `web/shell/i18n/core.{it,en}.json` | Shared OS vocabulary (Save, Cancel, Open, File, …). Merged **under every namespace**, so any app gets it for free. |
| `web/shell/i18n/shell.{it,en}.json` | The shell chrome (Start menu, taskbar, dialogs, toasts, context menus). |
| `apps/<id>/www/i18n.{it,en}.json` | Per-app catalog. Co-located with the app; resolved automatically as `/apps/<id>/i18n.<lang>.json`. |
| `tools/i18n-check.mjs` | The gate (`npm run i18n:gate`): JSON validity, key-set parity across languages, plural shape, placeholder parity, and that every `data-i18n` key referenced in HTML exists. |

### The language signal

The active language is the existing OS-wide key `localStorage['anima.lang']` (`'it' | 'en'`). It is
mirrored from the canonical store `/system/config/settings.json → ui.language` by the shell on every
load, and written there by the Settings picker. The engine reads `anima.lang`, falling back to
`<html lang>`, then `navigator.language`, then the base language `it`.

### Live, OS-wide switching (no reload)

Changing the language in **Settings** (or the copilot toggle, or onboarding) writes `anima.lang`.
That fires a `storage` event in **every other same-origin document** — i.e. all the open app iframes
— and the engine in each repaints itself. The setting frame and the shell are updated directly via
`setLang()` + a `set-language` postMessage. Net effect: pick a language anywhere and the whole OS,
including every open window, switches instantly. `<html lang>`/`dir` are kept in sync for CSS and a11y.

## Using it in an app

The smallest correct integration is **annotate the HTML + one bootstrap line + two catalog files**.

1. Annotate static markup (declarative — the engine fills these, and a MutationObserver also
   translates any annotated element the app adds later, so you rarely call `apply()` yourself):

   ```html
   <h1 data-i18n="title">Titolo</h1>
   <input data-i18n-attr="placeholder:search_ph;title:hint" placeholder="Cerca">
   <p data-i18n-html="rich">…</p>                 <!-- innerHTML, for trusted catalog strings -->
   <span data-i18n="left" data-i18n-args='{"n":3}'>…</span>   <!-- interpolation vars -->
   ```
   If an element has an icon **and** a label, wrap only the label: `<button><svg…></svg><span data-i18n="x">…</span></button>`.

2. Bootstrap once. If the app's main script is a module, import the engine and use the bound `t`:

   ```js
   import I18N from '/nucleo-i18n.js';      // absolute, NO /www/
   const t = await I18N.init('myapp');      // loads core + myapp catalogs, paints, observes, live-switches
   el.textContent = t('greeting', { name }); // dynamic strings built in JS
   I18N.onChange(() => repaintImperativeBits());
   ```
   If it is **not** a module, add a self-contained module near the end of `<body>` — no coordination needed:

   ```html
   <script type="module">import I18N from '/nucleo-i18n.js'; await I18N.init('myapp');</script>
   ```
   and annotate the strings inside your JS `innerHTML` templates with `data-i18n` so the observer
   translates them on insertion.

3. Write `apps/myapp/www/i18n.it.json` and `i18n.en.json` with the **same key set** (the gate enforces
   it). Don't redefine generic words — reference `core` keys (`save`, `open`, `cancel`, `file`, …)
   directly; only add keys for app-specific text.

### `t()` semantics

- Lookup order: app namespace → `core` → the key itself (never blank; a visible raw key flags a gap).
- `t('items', { count })` selects a plural form via `Intl.PluralRules` when the value is
  `{ "one": "…", "other": "…" }`.
- `{name}` placeholders are interpolated from `vars`.

### Locale-correct formatting

Use the engine's `Intl` helpers instead of hand-formatting, so dates/numbers follow the language:
`I18N.fmtDate(d)`, `fmtTime`, `fmtDateTime`, `fmtNumber(n)`, `fmtBytes(n)`, `fmtRelative(-3,'minute')`,
`fmtList([...])`.

## Adding a language

1. Add a row to `LANGS` in `web/shell/nucleo-i18n.js` (`{ code, label, flag, dir, locale }`). The
   Settings and onboarding pickers render from this list, so the option appears automatically.
2. Add `core.<lang>.json` and `shell.<lang>.json`, then `apps/<id>/i18n.<lang>.json` per app.
3. `npm run i18n:gate` until green. (Update `LANGS`/`BASE` in `tools/i18n-check.mjs` to match.)

RTL is anticipated: set `dir: 'rtl'` in the `LANGS` row and the engine flips `<html dir>`.

## Build & deploy notes

- Catalogs and the engine are precached by the service worker (`web/shell/sw.js`) so the chrome is
  translatable offline. **Bump the `CACHE` version in `sw.js` whenever a shell asset changes**, or the
  browser serves the stale cached copy (this is the #1 "my change didn't show up" trap).
- Regenerate `.gz` siblings after editing served assets: `node tools/gzip-assets.mjs web/shell apps`.
- The simulator (`node tools/serve-shell.mjs`, port 5599) serves all of this exactly like the device
  (`/<asset>` → `web/shell`, `/apps/<id>/x` → `apps/<id>/www/x`).

## Out of scope (follow-up)

The **native on-device UI** (the C launcher/apps in `firmware/`) is a separate track: it would read
`ui.language` and use a compile-time string table keyed by enum. It requires a firmware flash and is
not covered here. The web PWA is the primary interface, so it is translated first.
